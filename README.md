## Возможности

### Голосовые заметки
- Отправляешь голосовое сообщение → бот транскрибирует локально через **mlx-whisper** 
- Показывает транскрипцию с кнопками **Подтвердить** / **Редактировать**
- После подтверждения сохраняется в БД; если редактируешь — бот ждёт текст и сохраняет исправленную версию
- Все служебные сообщения (транскрипция, промпт редактирования, ответ пользователя) удаляются после сохранения — чат остаётся чистым

### Недельный план
- `/plan` — генерирует структурированный план из всех голосовых за текущую неделю через **Claude claude-opus-4-6** (с extended thinking)
- Если план уже есть и новых голосовых нет — просто показывает из БД
- Если появились новые голосовые — добавляет задачи к существующему плану, не трогая старые
- Выполненные задачи отображаются **зачёркнутыми** — рендерится динамически, `plan_html` в БД не меняется
- `/re_plan` — полностью перегенерирует план из тех же голосовых

#### Блоки плана
| Эмодзи | Категория |
|--------|-----------|
| 👨‍💻 | Инженерные задачи |
| 🤝 | Тимлидские задачи |
| 🚀 | Развитие |
| 📚 | Саморазвитие |
| 🗂 | Организационные моменты |
| 📌 | Другое |

### Рабочий день (`/work`)
- Показывает все невыполненные задачи из плана в виде нумерованного списка
- Задачи, уже выполненные на этой неделе, не показываются
- Пользователь выбирает задачи на сегодня (по умолчанию ничего не выбрано), порядок — как в плане
- После нажатия **Начать день** задачи показываются по одной с кнопками:
  - **✅ Сделал** — помечает задачу выполненной, переходит к следующей
  - **⏭ Отложить** — циклически переходит к следующей pending-задаче, текущая остаётся pending
- Когда все задачи выполнены — показывает итоговый прогресс
- Если вызвать `/work` повторно в тот же день — продолжает с того места где остановился

### Управление рабочей сессией
- `/re_work` — показывает задачи из плана, которых ещё нет в сегодняшней сессии; позволяет выбрать что добавить
- `/clear_work` — сбрасывает сессию текущего дня полностью (включая выполненные задачи); после этого `/work` начинает день заново

### Еженедельный отчёт
- `/report` — Claude анализирует что выполнено и что пропущено за неделю, даёт честную обратную связь

### Бэклог
- `/add_backlog` — переводит бота в режим ожидания голосового; голосовое транскрибируется и добавляется в бэклог (не в план недели)
  - Поддерживает редактирование перед сохранением
  - После подтверждения режим снимается — следующее голосовое идёт в обычный поток
- `/backlog` — показывает список pending-задач бэклога с кнопками:
  - **✅ N** — отмечает задачу выполненной
  - **🗑 N** — удаляет задачу

## Архитектура

```
Telegram User
     │
     ▼
aiogram 3.x (async polling)
     │
     ├── handlers/voice.py      — голосовые → транскрипция → подтверждение
     ├── handlers/commands.py   — /start, /plan, /re_plan, /history
     ├── handlers/work.py       — /work, /re_work, /clear_work, /report
     ├── handlers/backlog.py    — /add_backlog, /backlog
     │
     ├── services/transcription.py  — mlx-whisper (локально, M1 GPU)
     ├── services/planner.py        — Anthropic Claude API
     └── services/storage.py        — aiosqlite (SQLite)
```

### Порядок регистрации роутеров

```python
dp.include_router(commands.router)        # /plan, /re_plan, /history, /start
dp.include_router(work_handler.router)    # /work, /re_work, /clear_work, /report
dp.include_router(backlog_handler.router) # /add_backlog, /backlog — ДО voice!
dp.include_router(voice.router)           # все остальные голосовые
```

Порядок важен: `backlog_handler` должен стоять перед `voice`, чтобы перехватывать голосовые когда активен `BacklogStates.waiting_for_voice`.

### FSM состояния

```
TranscriptionStates.waiting_for_edit   — ожидание исправленного текста транскрипции
PlanStates.waiting_for_edit            — ожидание исправленного плана
WorkStates.selecting                   — выбор задач для /work и /re_work
BacklogStates.waiting_for_voice        — ожидание голосового для бэклога
BacklogStates.confirming               — подтверждение/редактирование транскрипции бэклога
BacklogStates.waiting_for_edit         — ожидание исправленного текста для бэклога
```

### Используемые модели Claude

| Функция | Модель | Thinking |
|---------|--------|----------|
| Генерация плана | claude-opus-4-6 | enabled, budget 2000 tokens |
| Добавление задач в план | claude-opus-4-6 | enabled, budget 2000 tokens |
| Еженедельный отчёт | claude-opus-4-6 | enabled, budget 1000 tokens |

## База данных

SQLite, файл `data/bot.db`. Путь задаётся через `DB_PATH` в `.env` или вычисляется автоматически относительно корня проекта.

Режим WAL включён (`PRAGMA journal_mode=WAL`) для лучшей производительности при конкурентном доступе.

### Таблица `transcriptions`

Хранит все подтверждённые транскрипции голосовых заметок.

```sql
CREATE TABLE transcriptions (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id    INTEGER NOT NULL,
    file_id    TEXT    NOT NULL,   -- Telegram file_id
    text       TEXT    NOT NULL,   -- транскрибированный текст
    duration   INTEGER,            -- длительность аудио в секундах
    created_at DATETIME DEFAULT (datetime('now'))
);
```

**Добавляется:** при подтверждении транскрипции (кнопка ✅ или после ручного редактирования).

**Не удаляется** — хранится бессрочно как история заметок.

**Используется:** командой `/plan` — берёт все записи за текущую неделю (`created_at >= week_start`).

---

### Таблица `task_plans`

Один план на пользователя на неделю (уникальный индекс по `user_id + week_start`).

```sql
CREATE TABLE task_plans (
    id                INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id           INTEGER NOT NULL,
    week_start        TEXT    NOT NULL,   -- дата понедельника: 'YYYY-MM-DD'
    transcription_ids TEXT    NOT NULL,   -- JSON-массив id транскрипций, вошедших в план
    plan_html         TEXT    NOT NULL,   -- HTML-разметка плана для Telegram
    created_at        DATETIME DEFAULT (datetime('now')),
    updated_at        DATETIME DEFAULT (datetime('now'))
);
```

**Добавляется / обновляется** через `INSERT OR REPLACE` при:
- Генерации нового плана (`/plan` без существующего)
- Добавлении новых задач к плану (`/plan` при наличии новых голосовых)
- Полной перегенерации (`/re_plan`)
- Ручном редактировании плана через кнопку ✏️

**`transcription_ids`** используется для определения "новых" голосовых: при следующем вызове `/plan` сравниваются текущие ID транскрипций недели с сохранёнными — разница идёт в `add_to_plan()`.

**`plan_html`** содержит чистый HTML (без зачёркиваний). Зачёркивания выполненных задач рендерятся динамически при показе через `render_plan_with_done()`.

**Не удаляется** — по завершении недели план становится недоступен (новый `week_start`), но физически остаётся в БД.

---

### Таблица `work_sessions`

Одна сессия на пользователя на день (уникальный индекс по `user_id + date`).

```sql
CREATE TABLE work_sessions (
    id                      INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id                 INTEGER NOT NULL,
    date                    TEXT    NOT NULL,   -- 'YYYY-MM-DD' по московскому времени
    tasks                   TEXT    NOT NULL DEFAULT '[]',  -- JSON-массив задач
    display_idx             INTEGER NOT NULL DEFAULT 0,     -- текущая показываемая задача
    current_task_message_id INTEGER,  -- message_id последнего сообщения с задачей
    created_at              DATETIME DEFAULT (datetime('now'))
);
```

**Структура элемента в `tasks`:**
```json
{
  "text": "Провести 1:1 с командой",
  "category": "tl",
  "emoji": "🤝",
  "status": "pending"
}
```

Возможные значения `status`: `pending`, `done`.

**Добавляется:** при нажатии "Начать день" в `/work` — создаётся запись с выбранными задачами.

**Обновляется:**
- `tasks` — при нажатии ✅ Сделал (статус → `done`) или добавлении задач через `/re_work`
- `display_idx` — при каждом переходе к следующей задаче
- `current_task_message_id` — при каждой отправке нового сообщения с задачей

**Удаляется:** командой `/clear_work` — полный сброс сессии текущего дня.

**Используется:**
- `/work` повторно — продолжает с `display_idx`
- `/report` — собирает все done-задачи за неделю
- `/plan` — выполненные задачи зачёркиваются в плане
- Следующий `/work` — уже выполненные задачи не показываются в списке выбора

---

### Таблица `backlog_items`

Долгосрочные задачи, не привязанные к неделе.

```sql
CREATE TABLE backlog_items (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id    INTEGER NOT NULL,
    text       TEXT    NOT NULL,
    status     TEXT    NOT NULL DEFAULT 'pending',   -- pending | done
    created_at DATETIME DEFAULT (datetime('now'))
);
```

**Добавляется:** через `/add_backlog` + голосовое (после подтверждения транскрипции).

**Обновляется:** нажатие ✅ N в `/backlog` меняет `status` → `done`.

**Удаляется:** нажатие 🗑 N в `/backlog` — физическое удаление строки.

**`/backlog` показывает только `status = 'pending'`** — выполненные задачи из списка пропадают, но физически остаются в БД.

## Команды и сценарии использования

### Сценарий 1: Создание недельного плана

```
[Пн утром]
1. Отправить несколько голосовых с задачами на неделю
   → Бот транскрибирует каждое, показывает текст с кнопками
   → Подтвердить или отредактировать каждое
2. /plan
   → Claude генерирует структурированный план по блокам
   → Показывается с кнопкой ✏️ Редактировать
```

### Сценарий 2: Добавление задач в существующий план

```
[Середина недели]
1. Отправить голосовое с новой задачей → Подтвердить
2. /plan
   → Бот видит новые транскрипции, вызывает add_to_plan()
   → Claude добавляет только новые задачи, не трогая старые
```

### Сценарий 3: Рабочий день

```
[Утром]
1. /work
   → Список невыполненных задач недели (выполненные скрыты)
   → Отмечаешь что берёшь сегодня → Начать день
2. Задачи показываются по одной:
   → ✅ Сделал — переходит к следующей
   → ⏭ Отложить — пропускает, вернётся по кругу
3. Когда все выполнены → итоговый прогресс
```

### Сценарий 4: Добавление задачи в текущий день

```
1. /re_work
   → Задачи из плана, которых нет в сегодняшней сессии
   → Выбираешь нужные → Добавить
```

### Сценарий 5: Бэклог

```
1. /add_backlog → "Записывай голосовое"
2. Отправить голосовое
   → Транскрипция с кнопками: Добавить в бэклог / Редактировать
3. /backlog → список с кнопками ✅ и 🗑
```

### Полный список команд

```
/start          — приветствие и список команд
/history        — все голосовые заметки за текущую неделю

/plan           — показать/обновить недельный план
/re_plan        — полностью перегенерировать план

/work           — начать или продолжить рабочий день
/re_work        — добавить задачи в текущий день
/clear_work     — сбросить сессию текущего дня
/report         — итоги недели от Claude

/add_backlog    — добавить задачу в бэклог голосом
/backlog        — просмотр и управление бэклогом
```

## Установка и запуск

### Требования
- Python 3.12+
- Apple Silicon (M1/M2/M3) — для локальной транскрипции через mlx-whisper
- Telegram Bot Token ([@BotFather](https://t.me/BotFather))
- Anthropic API Key

### Установка

```bash
git clone git@github.com:Bsshevchenko/TaskPlanBot.git
cd TaskPlanBot
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### Настройка `.env`

```bash
cp .env.example .env
```

```bash
TELEGRAM_BOT_TOKEN=your_bot_token
ANTHROPIC_API_KEY=sk-ant-...
ALLOWED_USER_ID=123456789        # Telegram user ID, получить через @userinfobot
# DB_PATH=data/bot.db            # опционально
# WHISPER_MODEL=mlx-community/whisper-large-v3-mlx  # опционально
```

### Запуск

```bash
python -m bot.main
```

При первом запуске mlx-whisper автоматически скачает модель с HuggingFace (~1.5 GB).

## Нюансы и ограничения

**Привязка плана к неделе.** Неделя отсчитывается с понедельника по московскому времени. При смене недели старый план становится недоступен — нужно создавать новый. Невыполненные задачи не переносятся автоматически — их нужно добавить в бэклог через `/add_backlog`.

**Один пользователь.** `ALLOWED_USER_ID` ограничивает бота одним пользователем. Если не задан — бот отвечает всем, что нежелательно: транскрипция ресурсоёмка и запускается локально.

**Транскрипция в executor.** mlx-whisper запускается через `asyncio.run_in_executor` чтобы не блокировать event loop. На длинных аудио (>2 мин) может занять 10–30 секунд.

**Зачёркивание в плане.** Сопоставление выполненных задач происходит по точному совпадению текста. Если задача была вручную отредактирована в плане — зачёркивание может не сработать. `plan_html` в БД всегда хранится без тегов `<s>`.

**FSM в памяти.** Используется `MemoryStorage` — при перезапуске бота все FSM-состояния сбрасываются. Незавершённые флоу (ожидание редактирования и т.п.) прерываются.

**Конфликт нескольких экземпляров.** Telegram не позволяет двум процессам бота одновременно получать updates через polling. При запуске убедись, что предыдущий процесс завершён: `pkill -f "bot.main"`.
